# Demo Review Checklist — Ash

Guide for running the InspectReport MVP demo locally and collecting feedback from Ash’s first review.

## How to run locally

```bash
npm install
npm run dev
```

Open [http://localhost:3000](http://localhost:3000) in a modern browser (Chrome or Edge recommended for audio recording).

Verify the app builds cleanly:

```bash
npm run lint
npm run build
```

## Environment variable for real AI drafting

Copy `.env.local.example` to `.env.local` and set:

| Variable | Required | Purpose |
|----------|----------|---------|
| `OPENAI_API_KEY` | Yes, for real AI | Server-side observation draft generation |
| `OBSERVATION_DRAFT_MODEL` | No | Defaults to `gpt-4o-mini` |

Restart the dev server after changing env vars.

**Without `OPENAI_API_KEY`:** observation drafts still work using demo drafting logic. The UI shows: *“Using demo drafting because no AI key is configured.”*

**With `OPENAI_API_KEY`:** drafts are generated by OpenAI from field notes, transcripts, and observation metadata.

## Demo flow for Ash

1. **Dashboard** — Click **Try Sample Project** (or use an existing sample).
2. **Sample project** — Follow the on-screen **Demo walkthrough** checklist.
3. **Step 1** — Review the six sample observations (progress, deficiencies, follow-ups) with placeholder photos and notes.
4. **Step 2** — Edit an observation without draft text (e.g. a deficiency) and click **Generate Observation Draft**. Review and edit the generated description and recommended action.
5. **Step 3** — Click **Open Report Preview** to see Section A (Progress Summary) and Section B (Deficiencies).
6. **Step 4** — On the report page, click **Download Word Report** and open the `.docx` in Word.
7. **Step 5** — Add a photo or record/upload a voice note on any observation. Transcribe the audio (demo transcription) and optionally generate another draft.

## What is real vs demo/local

| Feature | Status |
|---------|--------|
| Project & observation CRUD | Real — stored in this browser only |
| Photos & audio attachments | Real — stored in this browser only |
| Sample project seed data | Demo — fictional project with placeholder images |
| Audio transcription | Demo — sample transcript text, editable |
| Observation draft (no API key) | Demo — rule-based drafting from field notes |
| Observation draft (with API key) | Real — OpenAI-generated text |
| Report preview assembly | Real — built from saved observation data |
| Word export | Real — generates a structured `.docx` from report data |
| Official H.H. Angus letterhead / `.dotx` template population | Not yet — preview approximates layout; export uses programmatic Word structure |
| Backend, auth, multi-user sync | Not in scope for this MVP |

A subtle footer on every page notes: *“Demo data is stored in this browser only.”*

## Known limitations

- Data does not sync across devices or browsers; clearing site data removes projects.
- Transcription is not speech-to-text — it inserts editable sample text for demo purposes.
- Report preview letterhead is a placeholder; final branded template integration is planned.
- Photo appendix vs inline photo placement follows current export settings; not yet configurable in the UI.
- Only the Site Observation Report template is implemented.
- No due diligence or memorandum report types yet.
- Deleting a project permanently removes associated media from browser storage.

## Suggested questions for Ash after review

Use these to guide the conversation:

1. **Report structure** — Does the report structure match what you need for a typical Site Observation Report?
2. **Observation wording** — Does the AI-assisted (or demo) observation wording save time compared to writing from scratch?
3. **Form fields** — Are the fields on the project and observation forms enough for your workflow? What is missing?
4. **Photo placement** — Is photo placement in the preview and Word export acceptable?
5. **Inline vs appendix** — Would you prefer inline photos in the body, appendix photos, or a mix depending on report section?
6. **Real site visit** — What would prevent you from using this on a real site visit today?
7. **Next report type** — Which report type should come next: Site Observation, Due Diligence, or Memorandum?
8. **Draft review** — Is the “AI-assisted draft — review before issue” flow clear enough for professional sign-off?
9. **Action register** — Is the action summary table useful, or should it match a different internal format?
10. **Demo clarity** — Did the sample project and walkthrough make the product intent clear within the first few minutes?

## Pre-review smoke test

Before Ash’s session, confirm:

- [ ] Empty dashboard shows persuasive copy and **Try Sample Project** CTA
- [ ] Sample project creates successfully with observations and photos
- [ ] Observation cards show draft/media badges
- [ ] Draft generation works (with and without API key)
- [ ] Report preview renders with observations
- [ ] Word download succeeds
- [ ] No user-facing technical jargon on main screens
- [ ] `npm run lint` and `npm run build` pass
